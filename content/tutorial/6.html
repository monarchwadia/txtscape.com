Search your world

"Search txtscape for all pages that mention 'glowing gem'."

The agent calls search_pages and finds hits across multiple pages — the cave room, the bartender's knowledge, anywhere the gem appears. One search, full picture.

In a real project, this is how you'd find every decision that references your auth middleware, or every runbook that mentions the payment queue.

See connections

Tell your agent
"Show related pages for rooms/cave.txt."

The agent calls related_pages and finds cross-references — Grimjaw knows about the cave, the gem is mentioned in the cave, the forest connects to it. Pages reference each other naturally through their content.

See history

If you've been committing to git as you go (you should!), try:

Tell your agent
"Show the git history for rooms/cave.txt."

The agent calls page_history and shows when the cave was created and when it collapsed. Your project memory has its own version history — who changed what, and when.

Organize with concerns

As your pages grow, you can add a config.json to define the memory's taxonomy. Create .txtscape/config.json:

{
    "concerns": [
        {
            "folderName": "rooms",
            "label": "Rooms",
            "description": "Game world locations with descriptions, exits, and items."
        },
        {
            "folderName": "characters",
            "label": "Characters",
            "description": "NPCs with descriptions, dialogue, and knowledge."
        },
        {
            "folderName": "items",
            "label": "Items",
            "description": "Objects the player can find and interact with."
        }
    ]
}

Now the agent knows your project's knowledge structure. If it creates a page outside these folders, it'll see a warning — a gentle nudge to keep things organized.

What you've learned

txtscape is not a game engine. It's persistent project memory for AI agents.

In this tutorial, the "project" was a text adventure. The pattern works the same in a real codebase:

  • Architecture decisions — why you chose Postgres, how auth works, what you tried and rejected
  • Runbooks — how to deploy, what to do when the queue backs up, incident response steps
  • Learning logs — bugs you hit, patterns that worked, things that surprised you
  • Customer insights — who uses what, feature requests, pain points

The pattern is always the same: store knowledge in txtscape pages, let the AI read them. The agent doesn't start from scratch — it starts from everything you've recorded.
